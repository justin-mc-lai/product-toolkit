# Sprint 规划与敏捷管理

## 1. Sprint 基础概念

### Sprint 构成

| 阶段 | 说明 | 时长 |
|------|------|------|
| Sprint Planning |  sprint 计划会议 | 2-4 小时 |
| Daily Standup | 每日站会 | 15 分钟 |
| Sprint Review | sprint 评审 | 1-2 小时 |
| Sprint Retrospective | sprint 回顾 | 1-1.5 小时 |

---

## 2. Sprint Planning 模板

### 2.1 Sprint 目标设定

```markdown
## Sprint {N}: {Sprint Goal}

**Sprint 目标**: {一句话描述本次 Sprint 要达成的目标}

**时间**: {开始日期} ~ {结束日期}
**团队**: {团队名称}
**容量**: {可用人天数} 人天

---

### 2.2 Sprint 承诺

| 序号 | 用户故事 | 故事点 | 负责人 | 状态 |
|------|---------|-------|--------|------|
| 1 | US-001: xxx | 5 | @张三 | To Do |
| 2 | US-002: xxx | 3 | @李四 | To Do |
| 3 | US-003: xxx | 8 | @王五 | To Do |

**承诺点数**: {总点数}
**历史平均**: {历史平均点数}
**团队信心**: 🟢高 / 🟡中 / 🔴低
```

### 2.3 故事点估算

**斐波那契数列**: 1, 2, 3, 5, 8, 13, 21

| 复杂度 | 故事点 | 说明 |
|--------|-------|------|
| 很简单 | 1 | 简单修改，1小时内完成 |
| 简单 | 2 | 小功能，半天内完成 |
| 中等 | 3 | 一般功能，1天内完成 |
| 较复杂 | 5 | 复杂功能，需要2-3天 |
| 复杂 | 8 | 大功能，1周 |
| 很大 | 13 | 超大功能，多人协作 |
| 巨大 | 21 | 需要拆分 |

### 2.4 容量计算

```markdown
## 团队容量计算

| 成员 | 可用天数 | 系数 | 实际可用 |
|------|---------|------|---------|
| 张三 | 10天 | 0.8 | 8天 |
| 李四 | 10天 | 0.9 | 9天 |
| 王五 | 8天 | 0.7 | 5.6天 |

**总可用**: {22.6} 人天
**缓冲**: 20%
**Sprint 容量**: {18} 人天
```

```markdown
|-------|---------|---------|------|

```

### 7.2 预测下个 Sprint

```markdown
## Sprint N+1 预测

**平均 Velocity**: 21 点
**可用容量**: 20 人天 (考虑请假/培训)
**预测点数**: 18-22 点

**建议承诺**:
- 优先级 P0: 15 点
- 优先级 P1: 5 点 (弹性)
```

## 8. Definition of Done (DoD)

### 8.1 完成的定义

```markdown
## Definition of Done

- [ ] 代码已编写并通过 Code Review
- [ ] 单元测试覆盖率 > 80%
- [ ] 集成测试通过
- [ ] UI/UX 符合设计稿
- [ ] 文档已更新 (API文档/用户文档)
- [ ] 在测试环境验证通过
- [ ] 产品经理验收通过
```

### 8.2 用户故事完成标准

```markdown
## 用户故事完成标准

### US-{ID}: {标题}

**完成条件**:
- [ ] 代码开发完成
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] UI 验收通过
- [ ] 产品验收通过
- [ ] 文档更新完成
- [ ] 部署到测试环境
```

## 9. 常见问题

### Q: 故事点数估不准怎么办？

**解决**:
1. 使用参考故事点法（参考已完成的类似故事）
2. 估算时考虑不确定性，增加缓冲
3. 回顾会议分析估算偏差，持续改进

### Q: Sprint 中需求变更怎么办？

**解决**:
1. Product Owner 决定优先级
2. 变更放入下一个 Sprint
3. 紧急变更需团队共识，减少相应故事

### Q: 有人请假如何处理？

**解决**:
1. 提前计算容量时考虑请假
2. 紧急情况可调整 Sprint 目标
3. 团队互助完成承诺
