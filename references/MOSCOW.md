# MoSCoW 优先级法则详解

## 概述

MoSCoW 是一种需求优先级排序技术，常用于敏捷开发和产品管理中。

## 定义

| 优先级 | 含义 | 描述 |
|--------|------|------|
| **Must** | 必须有 | 上线阻塞项，P0，本期必须实现 |
| **Should** | 应该有 | 重要但非阻塞，P1，提升用户体验 |
| **Could** | 可以有 | 锦上添花，P2，本期可选实现 |
| **Won't** | 不会有 | 本期不实现，远期规划 |

## 使用场景

### 1. MVP 版本规划

在规划最小可行产品时，确定哪些功能是必须的：

```
MVP 核心功能（Must）:
- 用户注册/登录
- 核心业务流程
- 基础数据展示

增强功能（Should）:
- 高级搜索
- 消息通知
- 数据导出

锦上添花（Could）:
- 社交分享
- 个性化推荐
- 高级图表
```

### 2. 冲刺规划

在每个 Sprint 规划时，优先选择 Must 级别的用户故事：

```
Sprint 1 计划:
├── Must (必须完成)
│   ├── US-1: 用户登录
│   ├── US-2: 创建比赛
│   └── US-3: 比赛详情页
├── Should (尽量完成)
│   ├── US-4: 搜索功能
│   └── US-5: 筛选功能
└── Could (时间允许则做)
    └── US-6: 分享功能
```

### 3. 发布前需求冻结

在产品发布前，对需求进行最终排序：

```
发布检查清单:
✓ 所有 Must 级别功能已完成
✓ Should 级别功能完成率 > 80%
✓ Could 级别功能完成情况不影响发布
```

## 实施建议

### 1. 比例参考

- **Must**: 20-30% - 核心价值功能
- **Should**: 30-40% - 重要但非核心
- **Could**: 20-30% - 锦上添花
- **Won't**: 10-20% - 本期放弃

### 2. 决策原则

- **Must 决策**：产品负责人 + 技术负责人共同确认
- **Should/Could 决策**：产品负责人决定
- **Won't 决策**：产品路线图决定

### 3. 沟通技巧

向团队解释优先级：
- "这是 Must，因为没有它用户无法完成核心操作"
- "这是 Should，因为它能显著提升转化率"
- "这是 Could，它是nice-to-have但不是必须的"
- "这是 Won't，我们计划在下个季度实现"

## 与其他方法对比

| 方法 | 适用场景 | 复杂度 |
|------|---------|--------|
| MoSCoW | 快速排序、MVP规划 | 简单 |
| KANO | 用户调研、满意度分析 | 中等 |
| RICE | 多因素权衡 | 复杂 |
| ICE | 快速迭代决策 | 简单 |

## 示例：功能排序

```
功能清单原始列表:
1. 用户登录
2. 第三方登录
3. 社交分享
4. 消息推送
5. 数据导出
6. 离线缓存

MoSCoW 排序结果:
├── Must
│   ├── 用户登录（基础功能）
├── Should
│   ├── 第三方登录（提升注册转化）
│   └── 离线缓存（提升体验）
├── Could
│   ├── 社交分享
│   ├── 消息推送
│   └── 数据导出
└── Won't
    └── （本期无）
```
